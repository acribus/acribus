# 介绍

# Philosophy

  • 约定优于配置
  • 纯函数组件（store 是视图的 only trueth）
  • 开发页面时大多无需关注路由

# 用途

中后台项目通常服务于公司内部查看和配置数据，以支撑业务发展或提高效率。它的特点是以快速完成业务需求为目标，对兼容性、样式和页面布局要求较低。这种情况下组件级别的复用已经不能满足对效率的追求。

Acribus 用于助你在组件之上完成下一个效率提升。 如何对多样的中后台需求进行分类并对应到一个解决方案是关键， Acribus 确立了一种对需求分类的标准 —— 资源

# 什么是资源

每一个需求都可以看做是在操作某类或某几类数据，我们把一类数据称为一种资源。

# 为什么不是另一个页面素材库

  • 素材库使用上困难，怎样把一个需求和一个页面素材对应起来 ？
  • 可能存在多个素材通过配置实现相同的需求（解决方案重叠），
  • 理论上素材库永远只能是有限解决方案。

Acribus 明确了可操作的需求划分标准，并定义了每个类别的对外接口

# 它是如何工作的？

事实上 Acribus 只是一种分类方案及接口规范，具体实现上会基于几大框架和对应的优秀组件库。 比如基于 Vue 的实现使用了Vue Router、Vuex、Element-ui， 但是它并不需要你了解对应框架，可以拿来即用，但是如果熟悉相应框架将会更便于调试。

# 特性

  • 完整的配套工具链
  • 快速高效率
  • 多语言支持
